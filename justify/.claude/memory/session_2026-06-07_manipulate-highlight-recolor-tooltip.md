**STAGE 3 - PANEL OVERLAP - ROOT-CAUSED + FIXED in source (pending build+visual):** Jonah's screenshot showed pin-lines + handles painting OVER the panel. ROOT CAUSE: `.justify-container` (overlay.ts:22, the div inside the overlay shadow that holds the panel) is `position:fixed` but had NO z-index (auto). The picker chrome (highlight/selection/handles/labels/pin-lines) is appended as SIBLINGS of that container directly on the shadow root, with POSITIVE z-index (2147483644-2147483646). In the shadow root's stacking context, positive-z positioned siblings paint ABOVE a z-auto positioned element - so the picker chrome painted above the container, and `.retune-panel`'s z-index:2147483647 was useless because it only ranks WITHIN the container's own (fixed-position) stacking context, which itself sat below the picker chrome. The host's z-index:2147483647 (overlay.ts:14) is in the DOCUMENT context (vs page), a red herring. FIX (one line): added `z-index:2147483647` to `.justify-container`, lifting the whole container-context (panel included) above the picker chrome. Container is transparent + pointer-events:none so only its opaque child (the panel) covers anything; picker chrome on the page stays visible, only the part behind the opaque panel is hidden (the desired behavior). elementFromPoint still skips the pointer-events:none container so hover-detection is unaffected.
